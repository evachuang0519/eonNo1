────────────────────────────────────────────────────────────────
  [階段三] 資料庫正規化 — 2026-04-11
────────────────────────────────────────────────────────────────

  背景
  ────
  JSONB 架構雖彈性高，但無法做關聯查詢、缺乏資料完整性約束。
  決定將資料正規化，同時設計角色權限系統。

  設計決策
  ────────
  · 個案總表有「去程 / 回程」兩筆 → 拆分為 patients（人）+ patient_routes（路程）
  · 司機與司機名單（帳號）合併 → 一張 drivers 表
  · 帳號系統獨立：users 表關聯 roles / drivers / centers
  · 新增 dirty 旗標機制追蹤未推送變更
  · 新增 appsheet_key 欄位對應 AppSheet 主鍵

  資料庫 v2（schema_v2.sql）— 11 張資料表
  ──────────────────────────────────────────
    centers           日照中心        5 筆
    drivers           司機            8 筆
    patients          個案（人）      61 筆
    patient_routes    個案路程        122 筆（每人去程+回程）
    orders            班表訂單        612 筆
    roles             角色            4 種
    permissions       功能權限        20 項
    role_permissions  角色權限對應    44 筆
    users             使用者帳號      11 筆
    sync_queue        推送佇列        —
    sync_log          同步日誌        —

  角色定義
  ────────
    admin       系統管理員（全功能）
    dispatcher  調度員（班表 + 個案管理）
    center      日照中心（本中心資料）
    driver      司機（今日班表 + 行程更新）

  遷移執行（migrate_v2.py）
  ─────────────────────────
  ✅ Step 1：執行 schema_v2.sql（psql 子程序，繞過 pg8000 多語句限制）
  ✅ Step 2：日照名單 → centers（5 筆）
  ✅ Step 3：司機 + 司機名單 → drivers（8 筆，合併去重）
  ✅ Step 4：個案總表 → patients（61 人）+ patient_routes（122 路程）
  ✅ Step 5：日照班表 → orders（612 筆）
             修復：duration_min "0.0" 字串 → safe_int() 轉換
  ✅ Step 6：使用者帳號（5 中心 + 5 司機 + 1 admin = 11 筆）
  ✅ 驗證腳本（verify_migration.py）確認所有表格資料完整

  遷移驗證結果
  ────────────
    訂單有個案對應：82.5%（140 筆無對應，姓名不完全匹配）
    訂單有司機對應：100%

【已知問題 / 技術債】
================================================================

  優先度 HIGH
  ──────────
  ❶ Google Maps API Key 失效
     位置：backend/templates/map.html 最後 <script> 標籤
     症狀：地圖頁面顯示「糟糕！出了點狀況。」
     解法：至 Google Cloud Console 申請新金鑰，啟用 Maps JavaScript API
           更新 map.html 的 key= 參數
           建議：移至 config.py 統一管理（GOOGLE_MAPS_KEY）

  ❷ 訂單有 17.5%（約 140 筆）找不到對應個案
     位置：orders.patient_id IS NULL
     原因：AppSheet 遷移時姓名不完全匹配（可能有空格、全半形差異）
     解法：建立手動對應工具，或在 orders 中用 notes 欄位重新比對

  優先度 MEDIUM
  ─────────────
  ❸ 前端頁面（/today, /schedule, /cases 等）仍讀取舊版 JSONB API
     舊版 API 資料與正規化 v2 表格筆數不一致
     例：/api/司機 = 6筆，/api/v2/drivers = 8筆
     解法：逐步將頁面改接 /api/v2/* 路由

  ❹ 尚無登入驗證機制
     目前所有頁面直接開放，沒有身份驗證
     users 表與 roles 已建立，但未實作 session/token

  ❺ 軟刪除的記錄未推送至 AppSheet
     deleted=TRUE 的記錄從 dirty 計數排除
     需實作：偵測 deleted=TRUE + dirty=TRUE → 呼叫 delete_row

  優先度 LOW
  ──────────
  ❻ 遷移後 center IDs 從 6 開始（早期測試留下殘留值）
     不影響功能，但若需要從頭重建需清理

  ❼ patient_routes 預設司機覆蓋率只有 26.2%
     其餘 73.8% 無預設司機，可能影響自動產生班表品質

【技術決策記錄（ADR）】
================================================================

  ADR-001：以 Python Flask 取代 Classic ASP
  ──────────────────────────────────────────
  時間：2026-04-06
  決策：捨棄 IIS + Classic ASP，改用 Python Flask（Port 8000）
  原因：
    · ASP 維護困難，套件生態萎縮
    · Flask 可直接操作 PostgreSQL、呼叫 AppSheet API
    · 前後端合一（Jinja2），減少部署複雜度
  取捨：需要 Python 執行環境，但符合現有技術能力

  ADR-002：保留 AppSheet 作為行動端入口
  ──────────────────────────────────────
  時間：2026-04-06
  決策：不取代 AppSheet，改為本機系統與 AppSheet 雙向同步
  原因：
    · 司機端已習慣使用 AppSheet 行動版
    · AppSheet 有完善的離線支援
    · 本機系統負責管理、統計、複雜業務邏輯
  取捨：需維護兩端同步，增加複雜度

  ADR-003：同步方向改為推送（Pull → Push）
  ─────────────────────────────────────────
  時間：2026-04-11
  決策：移除每 30 秒自動拉取，改為手動推送
  原因：
    · 自動拉取會覆蓋本機的變更（資料競爭）
    · 本機是主資料來源，AppSheet 是展示 / 行動端
    · dirty 旗標提供更精準的同步控制
  取捨：AppSheet 端的變更不會自動同步到本機

  ADR-004：資料庫正規化（JSONB → 關聯式）
  ─────────────────────────────────────────
  時間：2026-04-11
  決策：將 JSONB records 表正規化為 11 張關聯式資料表
  原因：
    · JSONB 無法做 JOIN 查詢、外鍵約束
    · 個案資料結構複雜（人 + 多路程），需要拆表
    · 角色權限系統需要正規化結構
  取捨：需要遷移腳本，舊版 API 並行維護一段時間

  ADR-005：保留舊版 JSONB API 向下相容
  ─────────────────────────────────────
  時間：2026-04-12
  決策：/api/<table> 舊版路由繼續運作，新功能使用 /api/v2/*
  原因：
    · 現有前端頁面仍依賴舊版 API
    · 逐步遷移比一次性切換風險低
  計畫：等 Sprint 1 完成後逐步廢棄舊版路由
